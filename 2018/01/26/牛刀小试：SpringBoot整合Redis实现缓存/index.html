牛刀小试：Spring Boot整合Redis实现缓存

发表于 2018-01-26 | 分类于 Redis |

　　Spring 3.1 新增了一种全新的缓存机制，这种缓存机制与Spring容器无缝整合在一起，可以对容器中的任意Bean或Bean的方法增加缓存。Spring缓存不是一种具体的缓存实现方案，它底层需要依赖Redis、Guava等具体的缓存工具。因此应用程序只要面向Spring缓存API编程，应用底层的缓存实现可以在不同的缓存实现之间自由切换，应用程序无需任何改变，只要改变配置文件即可。
　　这里我将使用Spring Boot整合Redis实现缓存（ps：数据库使用MySQL，ORM框架使用JPA）。Redis是一个开源的，高性能的key-value数据库，即现在常说的NoSQL数据库。它可以用作数据库、缓存和消息中间件。Redis的优势包括它的速度、支持丰富的数据类型、操作原子性，以及它的通用性。

　　这里稍微说一下@Cacheable注解的用法吧（虽然感觉大家并不陌生）：
　　@Cacheable可用于修饰类或修饰方法。当使用@Cacheable修饰类时，Spring将在类级别上进行缓存——程序调用该类的实例的任何方法时都需要缓存，且共享同一个缓存区；当使用@Cacheable修饰方法时，Spring将在方法级别上进行缓存——只有当程序调用该方法时才需要缓存。
　　使用@Cacheable时可指定如下属性：
　　value：必需属性。该属性可指定多个缓冲区的名字，用于指定将方法返回值放入指定的缓存区中。
　　key：通过SpEL表达式显式指定缓存的key。若无该属性，缓存的key由刚刚配置的keyGenerator生成。
　　condition：该属性指定一个返回boolean值的SpEL表达式，只有当表达式返回true时，Spring才会缓存方法返回值。
　　unless：该属性指定一个返回boolean值的SpEL表达式，只有当表达式返回false时，Spring才会缓存方法返回值（其实condition和unless功能基本类似，但规则恰好相反）。

执行redisQueryTest()方法
　　控制台中并没有输出“–正在执行findByUserName()方法–”这句话，也没有输出hql语句（图就不贴了，因为…额…确实啥都没输出），说明findByUserName()这个方法根本没有执行，这是为什么呢？当传入方法的参数为hawk时，findByUserName()方法的key值与之前存入Redis的key值相同，这时将直接从Redis中获取数据，而不会执行相应的查询方法。这正是使用缓存的好处：对于需要经常查询的数据，可以省去频繁访问数据库的开销，提高效率。
