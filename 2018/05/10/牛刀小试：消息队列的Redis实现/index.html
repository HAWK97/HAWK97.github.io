牛刀小试：消息队列的Redis实现

发表于 2018-05-10 | 分类于 Redis |

　　最近闲着没事把博客的域名弄好了，大家以后可以直接通过http://wsghawk.com 来访问我的博客；还加上了畅言评论系统，有什么问题可以直接在评论里提（不知道为啥备案号一直过不了审，所以还是试用版本，难受）。
　　掐指一算，已经有一个多月没有更新博客了。一方面是因为最近在忙其他的事情，没时间（虽然这个“其他的事情”也没做好）；另一方面是感觉之前写的东西并没有帮到很多人，这让我……很犹豫。不过学技术从来都不是一蹴而就的，而是需要长时间的积累，想到这里，我觉得还是有必要继续把博客写下去。
　　之前写过一篇《入门消息队列》（http://wsghawk.com/2018/02/01/%E5%85%A5%E9%97%A8%E6%B6%88%E6%81%AF%E9%98%9F%E5%88%97/ ），结尾我说打算写一篇关于消息队列应用的文章。如今三个月过去了，觉得是时候把这个坑填上了！所以说这篇文章就来介绍一下消息队列的Redis实现。

　　在开始正式内容之前，先来回顾一下消息队列的两种模式：

  • 生产者消费者模式：生产者生产消息放到队列里，多个消费者同时监听队列，谁先抢到消息谁就会从队列中取走消息。即对于每个消息只能被最多一个消费者拥有。
  • 发布者订阅者模式：发布者生产消息放到队列里，多个监听队列的消费者都会收到同一份消息。即正常情况下每个消费者收到的消息应该都是一样的。

　　那为什么我要使用Redis作为消息队列中间件呢？大家经常听到的RabbitMQ、kafka等，多为企业级MQ产品，较为笨重。例如我们如果用RabbitMQ就必须为它搭建一个服务器，同时如果要考虑可用性，就要为服务端建立一个集群，而且在生产如果有问题也需要查找功能。在中小型业务的开发过程中，可能业务的其他整个实现都没这个重，过重的组件服务会成倍增加工作量。而Redis可以实现轻量级消息队列，适合我们平时开发使用。不过Redis设计的初衷并不是作为消息队列来使用，只是因为它的一些操作恰好可以实现上述的两种消息队列模式，实际性能与专业的消息队列产品肯定是有差距的。
　　说到Redis，五一三天精读了《Redis开发与运维》和《Redis实战》两本书（其实还有本《Redis设计与实现》，不过涉及的知识过于底层，没能读下去……），对Redis有了进一步的了解。它的效率之高，是MySQL这类传统关系型数据库无法比拟的。虽然对很多人来说，Redis还是一个新名词，但我觉得对于后台开发人员来说，Redis与MySQL一样，都是必须要掌握的。
　　这里我主要讲一下如何使用Redis实现消息队列的生产者消费者模式。该模式依赖于Redis的列表数据结构与lpush、brpop操作。大家都知道，lpush、lpop和rpush、rpop是在列表两端进行插入和弹出操作，而blpop和brpop是lpop和rpop的阻塞版本。简单解释一下，拿blpop来说，当列表内没有任何元素可供弹出的时候，blpop并不会像lpop一样立即返回一个空值，而是会将连接阻塞，直到等待超时或发现可弹出元素为止。将lpush与brpop操作结合到一起，就得到一个阻塞队列啦~
　　这里我主要结合最近项目中的一个业务来举例子：当用户点赞或评论其他用户的动态时，被点赞或评论的用户需要收到一条相关私信。我们在点赞时，其实主要关心这个赞是不是点上了，至于被点赞用户是否收到了相关私信，并不是那么重要（毕竟自己看不到），所以这个业务非常适合使用消息队列进行异步处理。
